# Roadmap — 2026-07-17

> Second pass: see docs/roadmap-2026-07-17-deepdive.md for unstated assumptions, options analysis, and pre-design specs for short-term items 1–5.

Prioritized short/medium/long-term plan for the human-moves-eval harness. Each item is tagged
**[TRACKER]** (existing owner-approved backlog, re-prioritized here) or **[NEW: source]**
(motivated by docs/research/LLM-Eval-1.md ("E1") or LLM-Eval2.md ("E2")).

## Current state

Six experiments are in the books, honestly mixed: exp1 was **inconclusive at ceiling** (rescored:
arms identical 18/20; the apparent baseline edge was a judge-scope artifact); exp1H off-ceiling was
a **dead heat** — combined, no detectable review-shape gain on the weak tier. Exp2's negative
control **validated true-negative** (harness_broken armed, correctly silent). Exp-d7 wrong-vs-smell
halved Sonnet-5's asserted-finding FPs (43%→22%) and is deployed as global steering. Exp-2's
verify-gate Stop hook is the headline positive: it **inverted** Fable's test-rigor edge on
implement work (12/12 unanimous). Exp-3 (2026-07-05) is the pivotal negative: **Fable's debug edge
survives the full steering stack even within-family — a capability gap, not a process gap** —
though D2 predict-then-probe still won 12/12 within-model on the hardest debug task. Harness
reality check (code, not docs): judge.py is a **single raw codex-only judge call** (no panel, no
bias correction, no claude provider); every item is **single-shot** (no pass^k / replicas — exp-3
and DES-05 both flag single-replay variance as a live caveat); judge-vs-human calibration is one
ad-hoc 20/20 spot-check with owner spotchecks still pending; scoring is already **binary
pass/fail + severity-aware WRONG/SMELL**, which matches the 2026 consensus (E2 §3 "Likert scales
are antipattern") — no scale change needed.

## Short-term (next 2–4 weeks) — cheap, high-leverage

1. **Claude-as-judge provider** [TRACKER] — judge.py is codex-only; required before any gpt-5.5
   executor run (guard otherwise forces an override). This is also the prerequisite for item 2.
2. **Two-family judge panel with disagreement escalation** [NEW: E1 §2, PoLL T1-18 + CyclicJudge
   T1-05] — panels of smaller judges beat a single strong judge at ~7× lower cost; the bench
   judging pipeline already runs dual blind judges, but harness/judge.py does not. Port the
   dual-family pattern into judge.py: 2 judges, agree ⇒ accept, disagree ⇒ third judge or human.
   Cheap because both CLI wrappers exist.
3. **Truth-set hygiene, forward-only** [TRACKER] — fix the rh-14 validity bug (the "clean" item
   contains a real Lock+depth defect; exp1H rh-14 fails may have been correct reviews), add the
   audited neutral_findings (rh-01/05/06/09/10/14), and the rh-06/rh-10 granularity/sibling items.
   This is the single biggest threat to result validity and costs no model calls.
4. **Judge accounting fixes** [TRACKER] — harden the codex judge-token parse (5/30 exp1H calls
   under-counted; totals are a floor), wire judge.usd_per_mtok rates end-to-end (rejudge.py
   judge_cost_usd currently stays None), and validate executor/judge effort enums at config load.
5. **Complete the pending owner spotchecks** [TRACKER, elevated by E1 §2A] — 3 split judging rows
   + the exp1h spotcheck file. Frame them as the start of a small **frozen human calibration set**
   (target 50–100 labeled findings): E1's converged recipe (calibration set → Rogan–Gladen or
   PPI++ correction → CI) needs exactly this asset, and nothing else on this roadmap can be
   bias-corrected without it. Report agreement as κ, not just percent-match (E1: r and κ jointly).
6. **Generalize adherence directive keys** [TRACKER] — hard-keyed to review-shape.*; blocks
   adherence-battery experiments (most queued moves are eval_shape: adherence).
7. **Drop ceiling items from active tasksets** [NEW: E2 §7 "Efficient Benchmarking", mid-difficulty
   filter] — INFRA-01 judged parity 12/12 ("too easy"); items outside a ~30–70% pass band burn
   budget without discriminating. One-line manifest edits, no new curation.

## Medium-term (1–3 months) — structural additions

1. **Replicas + pass^k for debug and design tasksets** [TRACKER "variance pass", upgraded by
   NEW: E2 §1 ReliabilityBench/Beyond-pass@1] — every exp-3/design cell is a single replay, and
   DES-05 showed all four fresh arms missing the vindicated direction on a replay. Given exp-3's
   capability-gap finding, the right next measurement on debug is not more steering arms but
   **characterizing the gap's shape**: k=3–5 replicas per cell, report pass^k and variance, and
   stratify by task difficulty/duration (reliability decays super-linearly with duration — E2).
   This tells you where the Opus+D2+hook substitute is *reliably* adequate vs. merely won-once.
2. **Bias-corrected metrics in metrics.py** [NEW: E1 §4 + T1-01/T1-02] — once the human
   calibration set from short-term #5 exists, add Rogan–Gladen point correction + Lang–Reiczigel
   CI (robust default) with PPI++/EIF as the efficient option. Judge-derived pass rates become
   defensible estimates instead of raw proxy counts. Small, self-contained addition next to
   wilson_ci/mcnemar_p.
3. **Unblock exp4 (rejected-alternatives on design) before exp3 (failure-signature on debug)**
   [TRACKER, re-ordered by the exp-3 result] — exp-3 shows debug is capability-limited, so a new
   debug-move ablation (queued exp3) has lower expected value than the design gap, where Sonnet 5
   collapsed (8/32) and moves are untested (rejected-alternatives is the catalog's declared
   must-test gap). Curate the design taskset + binary design-quality rubric first; keep the
   20-item reproducible-failing-test debug set as the *second* curation target — it doubles as the
   pass^k debug bank for #1.
4. **Judge stress suite** [NEW: E1 T1-06 RAND Judge Reliability Harness] — before scaling the
   panel, run cheap perturbation checks on review_judge.md: formatting/paraphrase invariance on a
   handful of graded findings blocks (formatting perturbations hurt judges more than semantic
   ones). Rerun on any judge model/rubric change.
5. **Taskset exploitability audit** [NEW: E2 §3 Berkeley RDI] — the IMPL-05 incident (judge ran
   `cat key.json`) already proved this class of failure locally; judge cwd isolation fixed the
   judge side. Audit the *executor* side the same way: can an arm score without solving (truth
   files reachable, oracle shortcuts, evidence gates gameable)? One session with the exploit
   mindset; document as PROVENANCE-style note per taskset.
6. **Trajectory/log-derived metrics from the transcript corpora** [NEW: E2 §2 TRACE/HAL Docent]
   — the reference transcripts are the program's durable asset. Adopt a Docent-style rubric
   (verification, self-correction, tool use, gaming) over existing transcripts to turn
   judged-endpoint comparisons into trajectory-level ones — this is also the instrument a
   capability-gap investigation needs (where in the trajectory does Fable pull ahead on debug?).

## Long-term (3–6+ months) — bigger bets

1. **Reframe the research question from "close the gap" to "certify the substitute"** [NEW: E1
   T1-03 Noisy-but-Valid; motivated by exp-3] — process gaps closed with hooks; the debug gap is
   capability and won't steer away. The durable product is therefore a **routing + certification
   harness**: for each work type, certify "config X's failure rate ≤ threshold" with
   finite-sample guarantees (Noisy-HT), rather than running ever-more move ablations chasing
   parity. Move ablations remain the tool for *within-model* wins (D2-style), not Fable-parity.
2. **Longitudinal drift bank across model versions** [NEW: E1 T1-10 Wiese blueprint; E2 §1
   drift] — capability gaps close when vendors ship new models, not when prompts change. Freeze
   the review-hard + debug + design banks and re-run them on each model-version transition
   (Sonnet 5→next, Opus 4.8→next, gpt-5.5→next), tracking pass^k and judge κ over time. This
   converts the July snapshot into a standing answer to "has the fleet caught Fable yet?" — and
   catches silent API drift (batch-invariance means temp-0 determinism is impossible on shared
   APIs; replicas + CIs are the mitigation, E2 §1).
3. **Contamination protection for the task banks** [NEW: E2 §1 DyePack; §3 watermarking shift] —
   the tasksets derive from private repos (low risk today), but any sharing or publication makes
   retrospective contamination detection impossible. If banks are ever shared, embed canary items
   first; meanwhile enforce the lineage rule (eval-authoring family ≠ judge family ≠ tested
   family — E2 Preference Leakage) and keep oracles history-anchored, never LLM-generated
   (synthetic eval items fail rank-preservation, E2 §1).
4. **IRT-based adaptive testing — only if the pool grows past ~100 items** [NEW: E2 §5 ATLAS/
   Trismik] — 90% item reduction at matched precision is real but needs an item bank an order of
   magnitude larger than today's (~30 scored items). Revisit after the medium-term curation
   lands; the nearer-term wins from that literature are already captured by the mid-difficulty
   filter (short-term #7) and bad-item detection (Hardy/Benchmark² — apply opportunistically
   during curation).
5. **c-CRAB-style test-driven review scoring** [NEW: E2 §1 c-CRAB] — long-term alternative oracle
   for review tasks: credit a review iff changes derived from it make auto-generated tests pass.
   Removes the judge from the loop entirely for the subset of findings that are executable;
   aligns with the repo's existing execution-verified truth philosophy. Expensive; prototype on
   5 review-hard items before committing.

## Explicitly not doing

- Numeric 0–5/0–10 judge scales: binary + WRONG/SMELL already matches the E2 consensus.
- More steering-stack arms on hard debugging chasing Fable parity (exp-3 settled it within-family).
- Retrospective contamination checks (MIA) on external benchmarks — not defensible per E2; we
  don't cite external benchmarks anyway.
