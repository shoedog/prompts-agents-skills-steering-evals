## The "evaluation-portion" adaptation (what replaces eval-methodology rigor)

For an LLM-eval harness, the rigor center of gravity is *eval methodology* — are the metrics valid, is the judge calibrated, is the test set uncontaminated. The faithful quant-platform analogue is **the joint rigor of (1) point-in-time / reproducibility verification and (2) fail-closed risk control** — but the sharp form is more specific than "test more." The eval-world question "is my metric measuring the capability I think it is?" maps to **"is my backtest measuring a tradeable edge, or a data artifact (look-ahead, survivorship, corporate-action leakage, backtest-vs-live drift)?"** The critical realization is that on this platform *determinism is already well-covered* (canonical digest pins, "no mocked databases," reproducible manifests) — but **determinism is not validity**: a backtest can be deterministically, reproducibly *wrong* if look-ahead is baked into the shared read path. So the quant-equivalent of eval rigor here is **validity-of-green and backtest-to-live fidelity**, not repeatability: proving the *absence* of leakage and the *fidelity* of the backtest→live contract, and ensuring the green checkmark corresponds to real-world correctness rather than to a self-consistent fixture. That is precisely where this codebase is thinnest (audits run only in backtest, determinism gates run only locally), and it is the lens I applied below.

## Section A — In-flight / concurrency / merge hazards (most urgent; specific to the branch right now)

### A1. Activation safety is enforced by *convention*, not *mechanism*, at the exact moment a second agent is mutating the loader
**Mechanism.** The whole A-3 activation state machine (`crates/tools/historical-backfill/src/canonical/activate.rs`) is correct *within a single process's happens-before*: pointer CAS on `(active_revision, generation)` under a `SELECT … FOR UPDATE` + a detached session advisory lock (`family_lock.rs` — verified to pin a dedicated connection, so no wrong-connection bug), with append-only activation events and crash-replayable reconciliation. But three of the invariants that keep it *safe* are enforced only by human/agent discipline and prose:
- "Flips are scheduled outside decision windows" (ADR-0036 D1) — an operational rule; nothing in code prevents a flip while a backtest or live decision is resolving.
- The revision-5 exact-resume requirement ("do not mark failed, allocate revision 6, activate, or flip"; ADR-0052/0055) lives in a **plan-doc live cursor**, not in a DB constraint. The advisory family lock serializes concurrent *activations at runtime*; it does **not** stop a second agent from editing the loader code, issuing an operator command that allocates revision 6, or merging a partial branch.
- Re-fetching under a new run id is forbidden because it "would move the frozen Monday source boundary and invalidate the source-bound identity, bar, action, GP1, and GP2 evidence" (ADR-0052) — i.e. the correctness of every gate rests on a *frozen external snapshot* whose provenance is protected by convention.

**Threatens:** Invariant 1 (PIT reproducibility of the entire historical dataset) and the integrity of the canonical data plane. **Evidence it's live:** the branch is literally in this half-finished, stateful, operator-interrupted condition now, with a concurrent editor. The revision-1 burn *already* died mid-way once (ADR-0052, type-confused digest comparison), proving the machine reaches partial states under real operation. **Severity: High · Likelihood: Medium (High if the branch is merged or advanced without honoring the cursor) · Status: the window itself is `known` (ADR-0036 D1); the merge/concurrent-agent-invalidates-frozen-evidence failure mode is `untracked`.** This is the single item I'd flag first.

### A2. Postgres control-plane and QuestDB data-plane can disagree because they are never written or read in one transaction
**Mechanism.** Serving is a cross-store operation with **no shared snapshot**: `resolve_family` reads the pointer (Postgres), `coverage_for` reads coverage (Postgres), `store::get_bars` reads bars (QuestDB over HTTP) — three independent reads across two engines (`crates/services/market-data-query/src/service.rs` get_bars). The pointer can therefore assert "revision R active" while QuestDB's materialization of R is incomplete. This is not hypothetical: **revision 3 failed its visibility barrier with 1,666 unapplied WAL transactions** (ADR-0053), i.e. the flip-readiness check caught QuestDB lagging the control plane. The mitigation (ADR-0053 WAL-fenced visibility barrier) is a *count*-based check ("second identical per-instrument count," ADR-0036) — which can pass while byte-content differs, and does not cover the *reverse* race introduced by ADR-0055 partition-pruning (a prune of an old revision's partitions racing a reader still pinned to that revision by a frozen selection). **Threatens:** Invariant 1 and Invariant 2 — a covered date with no materialized bar is a **silent gap** (survivorship/look-ahead-adjacent), returned as an absence rather than an error, because `visibility::filter_bars` treats "no row for a covered date" as simply empty. **Severity: High · Likelihood: Medium · Status: `partially-tracked` (WAL barrier exists; the count-vs-content and prune-vs-pinned-reader edge cases are untracked).**

### A3. Any consumer using ABSENT (default) selection silently binds to live-active and is exposed to the unverified window
**Mechanism.** In `policy.rs::resolve_pg`, the `DatasetSelection::Absent` branch returns `ResolvedFamilyPolicy::Active` straight from the pointer **without checking `market_data_serving_verification`**. Because the pointer flip commits in one transaction (`activate_candidate_inner`, tx.commit) and the verification row is inserted in a *separate later* transaction (`reconcile_activation_state`), an ABSENT read landing between those commits serves a **gated-but-unverified** revision that may be rolled back seconds later as G8-failed. The `event-backtest-engine` avoids this correctly — it resolves once via `GetServingSnapshot`, freezes `(version, generation, policy_id)` into `FrozenDatasetSelection`, and folds them into the run input digest (`replay.rs`, `engine.rs`). **But the ABSENT path still exists** (the legacy replay path uses `historical_dataset: None`), and *any* future or ad-hoc consumer that reads without pinning inherits both the unverified-window exposure and non-reproducibility (re-running after a flip yields different data; a run that resolved `@5#G` cannot be replayed after a rollback bumps the generation — the request now fails `HISTORICAL_DATA_GENERATION_MISMATCH`). **Threatens:** Invariants 1 & 2. **Evidence:** the two selection paths coexist in `replay.rs`; the ABSENT resolution has no verification gate by construction. **Severity: Medium-High · Likelihood: Medium · Status: `untracked` (the pin discipline is specified for backtests; the "ABSENT is a foot-gun for every other consumer" framing is not).**

## Section B — Point-in-time / look-ahead

### B1. Query-time corporate-action adjustment with a single window-end `as_of` back-adjusts earlier bars
**Mechanism.** `service.rs::execute_canonical_request(bars, actions, range.as_of_epoch_nanos)` applies all corporate actions with `ex_date ≤ as_of` to the whole returned series. If a caller requests a date *range* with `as_of = window_end`, a dividend/split whose ex-date falls *inside* the window retroactively adjusts bars *earlier* than the ex-date — so an earlier decision's adjusted price embeds knowledge of a later corporate action. Correct usage requires `as_of` = each decision instant, not the window end. **Threatens:** Invariant 1. **Evidence it's live:** the repo's own `layer-3-deferred-follow-ups.md (d)` already names this exact bar-side look-ahead; the adjustment code path is in the shared serving seam that both backtest and (future) live call. **Severity: High · Likelihood: Medium · Status: `partially-tracked` (named as a deferred follow-up; not yet closed, and it sits on the shared read path).**

### B2. Corporate actions have no knowledge-time (bitemporal) dimension — vendor restatements are retroactively "known"
**Mechanism.** `corporate_action.rs` filters actions by effective date only (`ex_date ≤ as_of`), keyed by `(dataset_family, dataset_revision)`. There is no `received_at`/ingestion watermark and no supersession-aware read, so when a vendor later *corrects* a split ratio or dividend, a backtest at a historical `as_of` sees the corrected value as if it were known at ex-date. The A-3 spec states this plainly ("Action PIT is not bitemporal vendor-knowledge history"). **Threatens:** Invariant 1 — this is a genuine look-ahead vector, not merely a missing feature: strategies that trade around corporate actions get a clairvoyant view of restatements. **Evidence:** `layer-3-deferred-follow-ups.md (a)` + the spec's explicit out-of-scope note. **Severity: Medium-High · Likelihood: Medium · Status: `partially-tracked` — reframe from "feature gap" to "PIT look-ahead vector" for the roadmap.**

### B3. The PIT primitive itself is triplicated and its historical publication-time is synthetic
**Mechanism (two coupled sub-risks).** (a) The as-of join — the atom of every PIT query — has **three independent implementations** (`finseries-rs::asof` in Rust, `polars.join_asof` in Python, `LATEST ON`/`ASOF JOIN` in SQL) reconciled only by a parity test. The correctness of the platform's central invariant rests on that one test's coverage; a semantic drift in any one implementation (tie-breaking, inclusivity at the boundary) is a silent PIT divergence between research and production. (b) For canonical *historical* vendor data, `publication_time` is a **fixed 2,700-second modeling convention, not an observed value** (ADR-0035 note in `time-semantics.md`). Any feature that keys off `publication_time` (as opposed to `observation_time`/`ingestion_time`) is reasoning over a constant, not a fact — and derived events (`feature.vector.v1`, `signal.forecast.v1`, `backtest.event.v1`) omit `publication_time`/`ingestion_time` entirely, so the "four timestamps on every record" invariant is not universal. **Threatens:** Invariant 1. **Severity: Medium · Likelihood: Medium · Status: `known` (both facts are documented) — nuance: downstream feature code must be audited to ensure it never treats synthetic publication_time as real, and the three-way as-of parity test's case coverage is the single point of PIT drift.**

## Section C — Backtest-vs-live contract drift

### C1. The safety *audits* run only in backtest; the live path shares contracts but not safeties
**Mechanism.** Invariant 2 guarantees identical *contracts/schemas/RPCs* between backtest and live (verified: shared crates, wiring-only broker/clock substitution, same `CheckPreTradeRisk` call). It does **not** guarantee identical *audits*. The leakage auditor, the determinism digest gate, and the survivorship/leaver handling all execute in the backtest host; the live decision path (`live-decision-runtime`) runs none of them. So a look-ahead or survivorship defect that the backtest auditor would catch can pass straight into live decisions undetected — the very class of bug the platform is architected to prevent is unguarded on the side where real money moves. **Threatens:** Invariant 2 (and, downstream, capital). **Evidence:** `platform-analysis-2026-06-12.md` states it directly ("Live path lacks the safeties the backtest has … the contracts are shared; the audits only run in backtest"). **Severity: High · Likelihood: Medium (High once live) · Status: `partially-tracked` — flagged only in an advisory doc that was never folded into an ADR (see F3).**

### C2. Reproducibility is a stated release-blocker but is enforced by a *local* step, not CI
**Mechanism.** Invariant 10 ("reproducibility is a release blocker") is operationalized as canonical digest pins — but the real-data determinism acceptance tests are `#[ignore]` and "enforced by the local determinism gate rather than normal CI today" (`05-backtesting.md`), and the nightly continuous-ops soak is explicitly non-required/non-gating. A change that breaks determinism can therefore merge green; the "block" depends on a human remembering to run the ignored tests on a host with QuestDB. **Threatens:** Invariant 10 / the credibility of every backtest result. **Severity: Medium-High · Likelihood: Medium · Status: `partially-tracked`.**

### C3. `signal.forecast.v1` has two producers guarded by a single parity test
**Mechanism.** The event catalog's "exactly one canonical producer" rule has one sanctioned exception: `signal.forecast.v1` is emitted by *both* `strategy-runtime` (backtest) and `live-decision-runtime` (live), bounded only by `live-decision-runtime/tests/parity.rs`. This is the highest-leverage drift point in the system — it is the precise seam where "same signal in backtest and live" is asserted rather than structurally guaranteed. Incomplete parity coverage means a strategy can backtest one way and trade another with no other tripwire. **Threatens:** Invariant 2. **Severity: High · Likelihood: Low-Medium · Status: `known` exception — nuance: treat parity-test *case coverage* as a first-class, versioned artifact, not an ordinary unit test.**

### C4. The fill/cost model is a placeholder that the docs describe as if it were finished
**Mechanism.** The shipped model is a single fixed 5 bps adverse, fill-only slippage (Layer 5c); the Phase-1 summary claims "spread + linear market impact + latency." The pre-trade risk gate also prices notional off the clean close while fills move (an accepted "sizing-vs-fill mismatch"). Strategies calibrated against this optimistic, frictionless-ish model can look economic in backtest and be underwater live once real spread/impact applies — the classic cost-model over-fit. **Threatens:** backtest→live economic fidelity (Invariant 2 in spirit). **Evidence:** the doc-vs-reality contradiction is in `05-backtesting.md` itself. **Severity: Medium (High as size grows) · Likelihood: High-once-live · Status: `partially-tracked`.**

## Section D — Data-revision & corporate-action correctness (the in-flight A-3 core)

### D1. The identity/digest layer is empirically fragile, and its correctness gate is an exact-count conservation law
**Mechanism.** The canonical-load slice has already produced ~24 ADRs of *edge-case dispositions* — reused tickers (`SWAG`/`TDACU`/`GIWWU`), shadowed/partial restatement overlaps (`SONDQ`, `WTMA/EMAT`), zero-dividend no-ops, scientific-decimal parsing, unmatched-listing episode truncation — each discovered against real archives, each a place the pipeline previously produced the wrong answer. The master correctness check is a brittle **exact-count conservation invariant** (`materialized + numeric no-ops + identity exclusions = raw eligible`, e.g. `558,598 + 11 + 2,087 = 560,696`). This fails *closed* (good — an unexpected count blocks the burn) but that same brittleness means **any** future vendor snapshot whose composition shifts by one row halts activation until a human authors a new disposition ADR. The revision-1 burn already failed on a *type-confused digest comparison* (raw SHA-256 vs domain-separated identity-binding digest, ADR-0052), showing the digest discipline itself is error-prone. **Threatens:** the canonical data plane's availability and, if a disposition is mis-authored, its correctness. **Severity: Medium-High · Likelihood: Medium · Status: `untracked` as an *aggregate* fragility observation (individual dispositions are each ADR'd; the meta-risk that this hand-curated conservation law does not scale to routine re-snapshots is not).**

### D2. The storage-headroom guard rests on a model the environment already violated, and retention is unimplemented
**Mechanism.** Activation requires ≥12 retained full revisions of headroom, gated by a capacity charge. The original charge premise (measure `df` free-space delta per revision) was **falsified in practice** — revision 4's free space *increased* by 3.68 GB (ADR-0054), so the guard was patched to a conservative charge. Meanwhile retention/eviction is **explicitly DEFERRED** (ADR-0036) — there is no implemented sweep, so the 12-revision headroom is a hard wall reached by accretion with no automated reclaim. A miscalibrated charge plus no eviction means the platform can wedge itself out of the ability to activate a corrected revision precisely when it most needs to. **Threatens:** data-plane availability / operational continuity. **Severity: Medium · Likelihood: Medium · Status: `partially-tracked`.**

## Section E — Fail-closed risk-check gaps

### E1. Only four of eight pre-trade controls are live, and several deferred ones are safety-critical
**Mechanism.** Live today: max order notional (fat-finger), gross leverage, single-name concentration, kill switch (global/account/strategy). **Follow-up (not built):** max position size, liquidity/%ADV cap, sector exposure, short-borrow, price band. The doc header says "eight controls" and "committed" while the status table says four — a self-contradiction that will mislead a go-live decision. Absent a liquidity/ADV cap and price band, a strategy bug can size into an illiquid name or chase a bad print with only notional/leverage as backstops. **Threatens:** Invariant 3. **Evidence:** `08-risk.md` status table vs phase summary. **Severity: High (as a live-gate) · Likelihood: N/A until live · Status: `partially-tracked` (walking-skeleton deepening layer 4) — nuance: the specific missing controls + the doc contradiction should be an explicit live go/no-go blocker.**

### E2. "Fail-closed" is defined only for *unreachable*, enforced per-caller, and limit-set activation is non-atomic
**Mechanism.** The default-deny rule is specified for the risk service being *unreachable* and is implemented independently by each caller — there is no central enforcement and no stated semantics for a service that is **reachable but hung, slow past the 50 ms p99, or returning an internal error**. A hung-but-connected risk service could stall the order path rather than cleanly deny. Separately, `risk_limits` "updates flow through a manual approval today; the mechanism for atomically activating a new limit set … needs spec" — a torn read across a limit-set change could evaluate an order against a mixed old/new limit set. **Threatens:** Invariant 3. **Severity: Medium-High · Likelihood: Medium-once-live · Status: `untracked` (the timeout/error-return semantics and the per-caller-divergence risk); limit-set atomicity is an open question in the doc.**

### E3. The broker-scope kill switch — the last-resort "stop everything at the venue" — does not exist
**Mechanism.** `EngageKillSwitch` returns `unimplemented` for `scope=broker`; only global/account/strategy scopes work, and those act on the platform's own order path. If the platform itself misbehaves (or an order escapes), there is no implemented control to halt at the broker. **Threatens:** Invariant 3 / capital. **Severity: Medium (High once real money) · Likelihood: Low · Status: `partially-tracked`.**

## Section F — Multi-agent process & operational readiness

### F1. The collaboration model has conventions for humans and no mechanism for concurrent agents on shared stateful artifacts
**Mechanism.** `AGENTS.md`/`agent-collaboration.md` cover style, allowlists, and a pre-flight checklist, and say "code review when more than one **human** is on the project." There is **no** provision for two agents concurrently mutating shared *stateful* artifacts — the canonical loader, the serving pointer, the frozen source snapshot, or this feature branch. The runtime family advisory lock protects concurrent *activations* only; it does nothing for concurrent code edits, concurrent operator commands, or honoring a plan-doc cursor. The live revision-5 exact-resume condition (A1) is the concrete instance: correctness depends on every agent reading and obeying a prose cursor that no lock enforces. **Threatens:** every invariant, via corrupted shared state. **Severity: High · Likelihood: Medium · Status: `untracked` — this is the specific process gap; the repo tracks *tooling* for agents (ADR-012) but not *concurrency/coordination* risk.**

### F2. The live path is operationally blind and disaster recovery is designed-but-not-implemented
**Mechanism.** The 10 live services run as `nohup cargo run` writing flat files, with no OTel/metrics on the live path, Loki in-memory (logs die with the container), no Alertmanager, and no per-alert runbooks (deferred to deepening layer 7). DR is documented (nightly `pg_dump`, QuestDB Parquet, "6-hour RTO to rebuild canonical bars from raw lake") but **no cron exists** and the RTO is aspirational. There is no operator-unavailability runbook ("if the operator is unreachable for N days, who/what flattens the book") and the live-money go/no-go rubric exists only as advisory prose. For a real-money system, the observability and recovery posture is the weakest link relative to the sophistication of the data-plane. **Threatens:** operational safety / capital preservation. **Severity: High-once-live · Likelihood: Medium · Status: `partially-tracked` in `platform-analysis` only.**

### F3. The platform's own self-assessment was never folded into authoritative tracking
**Mechanism.** `platform-analysis-2026-06-12.md` is explicitly "advisory, not an ADR," and its P1–P5 findings (strategy-validity void, no walk-forward embargo/purge in production, live-path blindness, unimplemented DR, unmodeled taxes/margin/PDT, missing post-trade reconciliation, secrets blast radius) were **not** promoted into `risks-and-decisions.md` or the phase docs. The delta between that analysis and the ADR log is itself an untracked seam — several of the sharpest known risks are recorded *nowhere authoritative*, so they will not surface in a normal roadmap pass that reads only the tracked docs. **Threatens:** roadmap completeness / governance. **Severity: Medium (meta) · Likelihood: High · Status: `untracked`.** For the roadmap writer: reconcile `platform-analysis` P1–P5 into the ADR log as a first, cheap, high-leverage action.
