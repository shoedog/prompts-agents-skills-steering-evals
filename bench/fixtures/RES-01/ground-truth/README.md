# Agent Knowledge Doc Cleanup Handoff

## Objective

Clean up the documentation in `/Users/wesleyjinks/code/agent-knowledge` so both humans and LLMs can get up to speed quickly on what is where, what is current, what is legacy, and which files are authoritative.

This is a documentation and information architecture task, not a runtime behavior change.

## Core Problem

The repo already contains most of the information needed for onboarding, but it is fragmented and some docs conflict with the live code. The biggest issue is not missing information. It is missing trust in which documents are current.

## Desired Outcome

After this cleanup:

- A new human contributor should be able to find the right entry point in under 2 minutes.
- An LLM should be able to identify the live architecture and avoid legacy paths without reading half the repo.
- The repo should clearly distinguish current runtime artifacts from retained historical/legacy materials.
- The documentation should prefer code-backed truth over narrative drift.

## Source of Truth

When docs and prose disagree, treat these as authoritative:

- `pipeline/src/main.rs`
- `pipeline/src/pipeline.rs`
- `pipeline/src/profile.rs`
- `pipeline/src/verdict.rs`
- `pipeline/src/models/finding.rs`
- `scripts/install.sh`
- `scripts/sync-kiro.sh`
- `policies/base.json`
- `config/user-config.template.json`

The current best existing bootstrap document is `LLM.md`, but it still contains some stale references and should not be assumed perfect.

## Confirmed Documentation Drift

### 1. Docs index is weak and partially stale

- `docs/README.md` is too thin to serve as the navigation hub.
- It still references `CONSOLIDATED_PLAN.md`, which does not exist.
- It does not help a reader decide which docs are canonical, historical, or audience-specific.

### 2. Architecture docs do not accurately describe runtime behavior

- `README.md` says Phase 6 is `R2 + 5 Cognitive Frames`.
- `docs/ARCHITECTURE.md` says `R2 + 4 cognitive frames`.
- The live code in `pipeline/src/profile.rs` shows profile-dependent behavior:
  - `quick`: no frames
  - `standard`: 2 frames (`bd`, `et`)
  - `thorough` / `critical`: 5 frames (`bd`, `bp`, `et`, `ca`, `od`)
- Verifier and judge are real runtime concepts in the code path and should be represented cleanly.
- Invoker backends (`kiro`, `bedrock`, `bedrock-tools`, `claude-code`) are part of the current CLI surface and should be documented.

### 3. Policy docs disagree with executable policy

- `docs/POLICIES.md` documents a smaller token budget than `policies/base.json`.
- `docs/POLICIES.md` also duplicates operational rules that should either be derived from code/config or clearly marked as narrative guidance.
- Label behavior should align with `pipeline/src/verdict.rs`.

### 4. Setup/config docs drift from generated config behavior

- `docs/SETUP.md` is generally good, but the manual setup and config details drift from `config/user-config.template.json` and `scripts/install.sh`.
- The config generated by `scripts/install.sh` differs from the template fields and examples, which is confusing for onboarding.

### 5. Evaluation docs describe commands that do not exist in the live CLI

- `docs/EVALUATION.md` documents `kiro review --eval --golden-set ...`.
- The actual CLI in `pipeline/src/main.rs` does not expose that interface.
- This doc should either be rewritten around the current tooling or explicitly marked historical/aspirational.

### 6. Requirements and risks docs contain outdated scope assumptions

- `docs/REQUIREMENTS.md` says non-code artifacts like Terraform, Helm, Docker are out of scope, but the repo clearly includes infrastructure steering and rubrics.
- `docs/RISKS.md` and `docs/troubleshooting.md` still contain old orchestrator-era framing in places.

### 7. Live vs legacy artifacts are not clearly separated

- `scripts/sync-kiro.sh` says orchestrator and diff-analyzer were removed in favor of the Rust pipeline.
- But `agents/orchestrator.json`, `prompts/orchestrator.md`, `agents/validator.json`, and related materials are still present in the repo.
- Historical documents in `docs/archive/` are fine, but current navigation does not make it obvious which old artifacts are retained only for reference.

7. Update or explicitly demote `docs/EVALUATION.md`.

Should answer:

- What this project is
- What binaries exist
- What the current pipeline roughly does
- Where to start for setup
- Where to start for architecture
- Where to start for LLM bootstrap
- Where legacy/reference material lives

Keep it short.

### docs/README.md

Purpose: canonical navigation hub.

- current runtime files

- legacy/reference material to ignore unless relevant

## Implementation Guidance

- Prefer small numbers of authoritative docs over many overlapping summaries.
- Prefer linking to canonical files over repeating detailed numbers in prose.
- If a document is mostly outdated, either:
  - update it to match live behavior, or
  - clearly label it as historical/reference material.
- Do not delete historical material unless there is a very strong reason.
- Do not change runtime code unless needed to fix obviously broken links or examples.
- Avoid inventing behavior that is not backed by code.
- If metrics are stale or unverified, either remove them from primary onboarding docs or clearly label them as dated snapshots.

## Acceptance Criteria

The task is done when all of the following are true:

- `README.md`, `docs/README.md`, and `LLM.md` no longer fight each other.
- There is an obvious, reliable reading path for both humans and LLMs.
- `docs/ARCHITECTURE.md` describes the live pipeline behavior at a high level without obvious contradictions to code.
- The repo clearly communicates what is current vs retained historical material.
- Primary onboarding docs do not reference nonexistent files or nonexistent CLI commands.
- Policy/evaluation/setup docs are either current or clearly scoped as non-canonical.

## Suggested Review Checklist

When the cleanup is ready, review it against this checklist:

- Did it reduce ambiguity, or just shuffle links around?
- Are all claims backed by code or config?
- Does it avoid over-documenting stale metrics as if they are current facts?
- Can a new contributor answer “where should I look to change X?” quickly?
- Can an LLM avoid reading legacy orchestrator-era materials unless explicitly needed?
